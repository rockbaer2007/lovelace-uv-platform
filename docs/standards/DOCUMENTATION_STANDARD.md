# Documentation Standard

## Purpose

Define the documentation rules and standards used throughout the
Lovelace UV Platform project.

A consistent documentation style improves readability, maintainability
and collaboration.

------------------------------------------------------------------------

## Scope

This standard applies to all documentation contained within the Lovelace
UV Platform repository.

Examples include:

-   README
-   Project Constitution
-   Vision
-   Architecture
-   Roadmap
-   Coding Guidelines
-   API Documentation
-   ADRs
-   Release Notes

------------------------------------------------------------------------

## Document Header

Every document shall begin with the standardized metadata header.

Required fields:

-   Project
-   Document
-   Document ID
-   Version
-   Release
-   Status
-   Language
-   Maintainer
-   Organization
-   Last Updated
-   License

------------------------------------------------------------------------

## Document Structure

Every document should follow this order:

1.  Header
2.  Title
3.  Purpose
4.  Related Documents
5.  Main Content
6.  Footer
7.  Revision History (recommended)

------------------------------------------------------------------------

## Document Status

Allowed values:

-   Draft
-   Review
-   Active
-   Deprecated

------------------------------------------------------------------------

## Versioning

Documentation evolves together with the software.

Whenever a document changes significantly, update:

-   Version
-   Release
-   Last Updated

------------------------------------------------------------------------

## Language

English is the primary documentation language.

German translations should:

-   use identical document IDs
-   follow the same structure
-   remain synchronized with the English version

------------------------------------------------------------------------

## Markdown Guidelines

Use:

-   ATX headings (#)
-   Short paragraphs
-   Bullet lists
-   Tables where appropriate
-   Code blocks for examples
-   Diagrams for architecture

Avoid:

-   Deep heading hierarchies
-   Long unstructured paragraphs
-   Mixing multiple topics in one document

------------------------------------------------------------------------

## Naming Conventions

Examples:

``` text
ARCHITECTURE.md
ARCHITECTURE.de.md

VISION.md
VISION.de.md
```

## Writing Principles

Documentation should be:

-   Clear
-   Concise
-   Accurate
-   Consistent
-   Maintainable

------------------------------------------------------------------------

## Guiding Principle

Documentation is part of the product.

Well-written documentation is considered as important as well-written
code.
